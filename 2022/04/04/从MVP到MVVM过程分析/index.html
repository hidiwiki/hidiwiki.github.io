    从MVP到MVVM过程分析

    一、基本概念

    经过前文的分析，我们知道MVP模式存在以下缺陷：

    • Presenter作为中间层，与View之间的通信大多数通过回调的方式，如果存在多个View共同监听同一个Presenter，会造成接口浪费

    • Presenter如果做耗时操作，在此期间，View可能被销毁，也即是Presenter无法感知View的生命周期

    • Presenter更新UI时不一定在主线程，需要转线程操作

    例如，构造一个音乐播放的app(模拟播放音乐)，基本画面如下

    使用MVP模式的App架构如下

    图中，View层包含Activity和一个悬浮窗Window（Activity模拟），向Presenter层注册回调以监听播放器状态，同步更新画面，其中接口Callback定义如下：

    public interface IPlayerCallBack {

    void onTitleChange(String title);

    void onProgressChange(int current);

    void onPlayerPlay();

    void onPlayerPause();

    void onCoverChange(String cover);
}

    View层收到这些回调后更新UI，同时用户操作View产生事件，通知Presenter层控制音乐播放，如暂停、上一曲、下一曲等，具体代码可以参考gitee仓库，tag为v1.0，基于以上的缺陷，我们进行逐步改进

    二、添加监听器

    消除接口冗余问题的一个思路是以UI进行驱动开发，UI需要什么数据，Presenter层就提供什么数据，各个画面根据需要去监听各自的数据，当数据发生变更时，更新UI，这里封装一个容器DataContainer方便UI进行数据的监听（仅用于演示，请忽略代码质量）

    public class DataContainer<T> {

    private List<IDataListener<T>> mListenerList = new ArrayList<>();

    public T mValue = null;

    public void setValue(T value) {
        mValue = value;
        for (IDataListener<T> listener : mListenerList) {
            listener.onChange(value);
        }
    }

    public DataContainer(T value) {
        mValue = value;
    }

    public DataContainer() {
        mValue = null;
    }

    public void addListener(IDataListener<T> l) {
        if (mListenerList.contains(l)) {
            return;
        }
        mListenerList.add(l);
    }

}

    其中，用到的接口IDataListener代码如下：

    public interface IDataListener<T> {
    void onChange(T value);
}

    使用时，只需要在Presenter中把数据放入对应的DataContainer中即可

    三、模拟耗时操作

    实际操作时，Model层请求数据不会在主线程中进行，而且也不会立即返回，如网络请求等，接下来模拟一个请求音乐列表的例子，并逐步解决存在的问题

    1. Model层负责请求网络，获取音乐列表
    public void loadMusicByPage(int page, int size) {
    List<Music> list = new ArrayList<>();
    new Thread(new Runnable() {
        @Override
        public void run() {
            for (int i = 0; i < size; i++) {
                String name = "music_" + i;
                String cover = "cover_" + i;
                String url = "url_" + i;
                list.add(new Music(name, cover, url));
            }
            mCallback.onSuccess(list);
        }
    }).start();
}

public interface OnLoadMusicResult {
    void onSuccess(List<Music> list);
    void onError(String msg, int code);
}

    请求的数据在Model层与Presenter层之间通过接口OnLoadMusicResult返回到Presenter层

    2. Presenter把请求的数据放入DataContainer中，供View层监听

    public DataContainer<List<Music>> mMusicList = new DataContainer<>();

public DataContainer<LoadState> mLoadState = new DataContainer<>();

    3. View层监听Presenter层中的数据（关心的数据），更新UI

    mMusicPresenter.mMusicList.addListener(new IDataListener<List<Music>>() {
    @Override
    public void onChange(List<Music> value) {
        String name = Thread.currentThread().getName();
    }
});

mMusicPresenter.mLoadState.addListener(new IDataListener<MusicPresenter.LoadState>() {
    @Override
    public void onChange(MusicPresenter.LoadState value) {
        Log.d("lhd", "Load state = " + value);
    }
});

    从上面的代码来看，从Model层返回的数据在子线程中，并不能更新UI，接下来我们先解决一个这个问题，修改DataContainer，完成线程的切换

    在DataContainer的setValue()方法中，做一下判断，如果当前线程是主线程，直接通知listener即可，如果不是，切换线程

    如何判断是不是主线程，通过以下方式

    private boolean isMainThread() {
    return Looper.getMainLooper().getThread() == Thread.currentThread();
}

    如何完成线程的切换？

    在Application中创建一个静态Handler，然后通过post方法切换线程

    public class App extends Application {

    public static Handler mHandler;

    @Override
    public void onCreate() {
        super.onCreate();
        mHandler = new Handler();
    }
}

    别忘了在Manifest中指定Application为自定义的App

    完成后的效果如下

    public void setValue(T value) {
    mValue = value;
    // 如果是主线程,直接回调
    if (isMainThread()) {
        for (IDataListener<T> listener : mListenerList) {
            listener.onChange(value);
        }
        return;
    }
    // 如果不是主线程，切换线程
    App.mHandler.post(new Runnable() {
        @Override
        public void run() {
            for (IDataListener<T> listener : mListenerList) {
                listener.onChange(value);
            }
        }
    });
}

    四、感知生命周期

    解决了接口冗余问题，线程切换问题，MVP模式目前还存在一个比较重要的问题，Presenter或Model层无法感知View层的生命周期，原因前文已经分析过了，在此不再赘述，本节我们来看一下如何解决这个问题

    4.1 ILifecycle接口

    将生命周期抽象出一套接口，然后Presenter层实现该接口

    public interface ILifecycle {
    void onCreate();
    void onStart();
    void onResume();
    void onPause();
    void onStop();
    void onDestroy();
}

    在MusicListActivity的各个生命周期中去通知Presenter，但是这样拓展性太差，如果另一个Activity也需要提供相应的生命周期，也需要这么做，因此，我们抽取出BaseActivity，在其中完成通知动作，其他activity继承BaseActivity即可

    public class BaseActivity extends AppCompatActivity {

    private List<ILifecycle> mLifeCycleListenerList = new ArrayList<>();


    public void addLifeCycleListener(ILifecycle lifecycle) {
        if (mLifeCycleListenerList.contains(lifecycle)) {
            return;
        }
        mLifeCycleListenerList.add(lifecycle);
    }

    @Override
    protected void onCreate(@Nullable Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        for (ILifecycle life : mLifeCycleListenerList) {
            life.onCreate();
        }
    }

    @Override
    protected void onStart() {
        super.onStart();
        ...
    }

    ...
}

    修改后的class图如下：

    这样做虽然Activity可以提供生命周期的通知，但是其他View则还是不能灵活的提供生命周期，接下来我们继续优化，LifecycleProvider出场

    4.2 LifecycleProvider

    LifecycleProvider用于管理所有的ILifeCycle，之前写在BaseActivity中的一些逻辑提取到LifecycleProvider中

    public class LifecycleProvider {

    private LifeState mCurrentLifeState;

    private List<ILifecycle> mLifeCycleListenerList = new ArrayList<>();

    public void addLifeCycleListener(ILifecycle lifecycle) {
        if (mLifeCycleListenerList.contains(lifecycle)) {
            return;
        }
        mLifeCycleListenerList.add(lifecycle);
    }

    public void unregisterLifecycleListener(ILifecycle lifecycle) {
        if (!mLifeCycleListenerList.contains(lifecycle)) {
            return;
        }
        mLifeCycleListenerList.remove(lifecycle);
    }

    public void updateLifeState(LifeState state) {
        if (state == LifeState.STATE_CREATE) {
            dispatchCreateState();
        } else if (state == LifeState.STATE_START) {
            dispatchStartState();
        } else if (state == LifeState.STATE_RESUME) {
            dispatchResumeState();
        } else if (state == LifeState.STATE_STOP) {
            dispatchStopState();
        } else if (state == LifeState.STATE_PAUSE) {
            dispatchPauseState();
        } else if (state == LifeState.STATE_DESTORY) {
            dispatchDestroyState();
        }
    }

    private void dispatchCreateState() {
        for (ILifecycle life : mLifeCycleListenerList) {
            life.onCreate();
        }
    }

    ...

}

    经过上面的改造，我们又发现了另外一个问题，Presenter的监听是在View中完成的，如下：

    public class MusicListActivity extends BaseActivity {
    
    ...
    
      @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_music_list);
        ...
        mMusicPresenter = MusicPresenter.getInstance();
        // mProvider在BaseActivity中创建
        mProvider.addLifeCycleListener(mMusicPresenter);
        ...
    }
}

    修改后的class图如下：

    也即是把整个Presenter全部注册给了LifeCycleProvider，这样做还是不够灵活，如果Presenter层的内部类需要监听View的生命周期，那就只能通过Presenter把生命周期进行转接给内部类了，而且Presenter类必须要实现ILifeCycle接口，有没有可能把注册的主动权交个Presenter呢？答案是肯定的，Presenter根据需要注册自己或其内部类或其他，ILifecycleOwner出场

    4.3 ILifecycleOwner接口

    具体方案如下

    • 将注册的过程移动到Presenter中进行，因此需要把LifeCycleProvider提供给Presenter，当然可以直接在创建Presenter时，把LifeCycleProvider直接传递给Presenter，这里采用一种适配性更好的方式

    • 提供一个ILifecycleOwner接口，让BaseActivity实现，用于Presenter提取LifecycleProvider，Presenter拿到ILifecycleOwner后，通过getProvider()获取LifeCycleProvider然后进行注册

    在此情况下，Presenter无法使用单例模式，后面再继续改进

    ILifecycleOwner的定义如下

    public interface ILifecycleOwner {
    LifecycleProvider getProvider();
}

    BaseActivity实现ILifeCycleOwner接口

    public class BaseActivity extends AppCompatActivity implements ILifecycleOwner {

    public LifecycleProvider mProvider = new LifecycleProvider();

    @Override
    public LifecycleProvider getProvider() {
        return mProvider;
    }
    ...
}

    子Activity继承BaseActivity，在创建Presenter时，将ILifecycleOwner传给Presenter

        public MusicPresenter(ILifecycleOwner owner) {
        mMusicModel = new MusicModel(mMusicResult);
        // Presenter将自己或其他内部类注册给Provider
        owner.getProvider().addLifeCycleListener(this);
    }

    4.4 让数据感知生命周期

    上面我们实现了让Presenter可以感知View的生命周期，下面我们把DataContainer也具备感知View的生命周期的功能，让View在用户可见的周期内更新UI

    改造DataContainer，与Presenter一样，在传入ILifecycleOwner，拿到LifecycleProvider，将自己注册到LifecycleProvider

        public void addListener(ILifecycleOwner owner, IDataListener<T> l) {
        // owner.getProvider().addLifeCycleListener(this);
        if (mListenerList.contains(l)) {
            return;
        }
        mListenerList.add(l);
    }

    如果这样做，会发现：

    • 有多个View监听该DataContainer，因此需要管理器ILifecycleOwner

    • DataContainer必须要实现ILifecycle接口，注册进LifecycleProvider中

    首先我们来解决第一个问题，在DataContainer中维护一个HashMap，以IDataListener<T>为Key，以ILifecycleOwner为Value

    private HashMap<IDataListener<T>, LifecycleProvider> mProviderMap = new HashMap<>();

public void addListener(ILifecycleOwner owner, IDataListener<T> l) {
    mProviderMap.put(l, owner.getProvider());
    if (mListenerList.contains(l)) {
        return;
    }
    mListenerList.add(l);
}

    进行判断：更新DataContainer时，从Provider中拿到当前的LifeState，如果大于STATE_START再更新

    public void setValue(T value) {
    mValue = value;
    if (isMainThread()) {
        for (IDataListener<T> listener : mListenerList) {
            dispatchValue(value, listener);
        }
        return;
    }
    App.mHandler.post(new Runnable() {
        @Override
        public void run() {
            for (IDataListener<T> listener : mListenerList) {
                dispatchValue(value, listener);
            }
        }
    });
}

private void dispatchValue(T value, IDataListener<T> listener) {
    LifecycleProvider provider = mProviderMap.get(listener);
    if (provider != null
            && provider.isAtLeast(LifeState.STATE_START)) {
        listener.onChange(value);
    }
}

    解决完上面的第一个问题，我们再来看一下第二个问题：

    DataContainer必须要实现ILifecycle接口，注册进LifecycleProvider中

    ILifecycle接口定义了七个生命周期，DataContainer大多数都用不到，为了更好的解决第二个问题，我们对ILifecycle改造一下，如下：

    public interface ILifecycle {
    void onViewLifeStateChanged(LifeState state);
}

    用一个抽象类AbsLifecycle实现ILifecycle接口

    public abstract class AbsLifeCycle implements ILifecycle {

    public void onCreate() {

    }

    public void onStart() {

    }

    public void onResume() {

    }

    public void onPause() {

    }

    public void onStop() {

    }

    public void onDestroy() {

    }

    @Override
    public abstract void onViewLifeStateChanged(LifeState state);
}

    之前使用ILifecycle接口的，现在换成AbsLifecycle抽象类

    在DataContainer中定义一个内部类继承AbsLifecycle，然后实现onViewLifeStateChanged方法，在state为Destroy时，清除对应的LifecycleOwner

    class DataListenerWraper extends AbsLifeCycle {

    private IDataListener<T> dataListener;

    public DataListenerWraper(IDataListener<T> l) {
        dataListener = l;
    }

    @Override
    public void onViewLifeStateChanged(LifeState state) {
        if (state == LifeState.STATE_DESTORY) {
            mProviderMap.remove(dataListener);
        }
    }
}

    addListener时将IDataListener<T> listener封装成DataListenerWraper，如下

    public void addListener(ILifecycleOwner owner, IDataListener<T> l) {
    mProviderMap.put(l, owner.getProvider());

    DataListenerWraper wraper = new DataListenerWraper(l);
    owner.getProvider().addLifeCycleListener(wraper);

    if (mListenerList.contains(l)) {
        return;
    }
    mListenerList.add(l);
}
